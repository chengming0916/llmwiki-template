## 工作流命令

### INGEST - 处理原材料 → 生成知识库页面
```
1. 读取 raw/xxx 文件
2. 在 wiki/sources/ 创建/更新源摘要页面（一一对应）
3. 提取关键概念 → 在 wiki/concepts/ 创建/更新概念页面
4. 识别实体 → 在 wiki/entities/ 创建/更新实体页面
5. 在相关页面之间添加双向链接
6. 更新 wiki/index.md 添加新条目
7. 在 wiki/log.md 记录操作
```

### QUERY - 基于知识库回答问题
```
1. 搜索 wiki/ 内容查找相关信息
2. 提供答案并标注引用的具体页面
3. 禁止编造不存在于知识库中的信息
4. 如信息不完整，说明情况并建议 INGEST
```

### LINT - 检查知识库完整性
```
1. 查找孤立页面（没有任何页面链接到它）
2. 查找断链（指向不存在的页面）
3. 检测页面间的信息矛盾
4. 查找重复内容
5. 报告发现的问题及具体修复建议
```

## 页面 YAML Frontmatter 标准

所有知识库页面必须包含以下 frontmatter：

```yaml
---
title: 页面标题
tags: [标签1, 标签2]
source: raw/xxx.md  # 仅用于源摘要页面；概念/实体页面省略或注明来源
created: 2026-xx-xx
updated: 2026-xx-xx
status: draft/reviewed
---
```

- 概念页：放在wiki/concepts/，讲清楚定义、原理、关联 
- 实体页：放在wiki/entities/，人物/工具/项目的信息、关系 
- 源摘要页：wiki/sources/，对应raw一篇，提炼核心、要点 
- 必须加**双向链接**，关联相关概念/实体

## 代码风格指南

### Markdown 格式规范
- 使用 ATX 风格标题（`#`、`##`、`###`）
- 无序列表使用短横线（`-`）
- 有序列表使用数字（`1.`）
- `**粗体**` 表示强调，`*斜体*` 表示次级强调
- `` `code` `` 用于技术术语、文件名、命令
- `> 引用块` 用于定义或引述
- 代码块标注语言：```python、```bash、```yaml 等

### 知识库内部链接
- 内部链接：`[[页面名称]]` 或 `[[页面名称|显示文本]]`
- 外部链接：`[文本](url)`
- 引用概念/实体时必须添加链接
- 避免孤立页面——每个页面至少被另一个页面链接

### 文件命名规范
- 使用中文名称，与内容匹配
- 不使用空格，需要时用 `-` 或 `_`
- 源摘要页面应与 raw 文件名一致

### YAML Frontmatter 规范
- 使用一致的标签名（先查看现有页面）
- 日期格式：`YYYY-mm-dd`
- 状态：`draft`（新建）、`reviewed`（已审核）

## 错误处理

### raw 文件缺失
- 在 log.md 报告：`WARNING: raw/xxx.md referenced but not found`
- 不创建源摘要页面

### 页面已存在
- 小心更新内容，保留 frontmatter
- 在 log.md 记录修改

### 链接目标不存在
- 如果逻辑上合理，创建目标页面（概念/实体）
- 如果不合理，报告为断链

## 导入约定

- 源摘要页面引用原始文件：`source: raw/notes/项目/文件.md`
- 概念/实体页面可引用多个来源
- 如有帮助，可在 frontmatter 中使用 `aliases` 添加别名

---

## 命名约定

- **标签**：小写、中划线分隔（如 `智慧校园`、`Qt开发`）
- **页面标题**：中文、描述性，除非是通用缩写否则不使用缩写
- **文件路径**：与页面标题一致，使用 `/` 分隔

---

## 内容质量规则

1. **禁止幻觉** - 只写原材料中存在的内容
2. **双向链接** - 如果 A 链接到 B，B 也应链接回 A
3. **术语一致** - 在所有页面中使用相同术语
4. **页面完整** - 每个页面应有实质性内容，不是空洞框架
5. **正确归因** - 始终引用原材料
